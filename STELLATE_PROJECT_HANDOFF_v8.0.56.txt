0. 문서 역할과 우선순위
======================================================================

이 파일은 새 채팅·담당자 변경·대화 소실 시 STELLATE 프로젝트를 이어가기 위한 최신 단일 인수인계 문서입니다.
앞으로 새 버전을 만들 때도 완료한 내용뿐 아니라 운영에서 추가 확인할 항목, 미확정 위험, 다음 버전에서 처리할 작업을 반드시 함께 기록합니다.

우선순위:
1) STELLATE_v8.0.56_FULL.zip 실제 소스
2) 이 인수인계서의 확정 정책
3) README_v8.0.56.md
4) 이전 버전 자료

실제 운영 Upstash Redis·QStash·Anthropic·Naver API에서 확인하지 않은 결과는 완료로 표시하지 않습니다.

1. 프로젝트 개요
======================================================================

서비스명: STELLATE
기존 저장소명: hotpick
도메인: stellate.co.kr
프레임워크: Next.js 14 Pages Router, React 18, Node.js 22.x
저장·작업: Upstash Redis, Upstash QStash
Redis namespace: stellate:v7

목적:
- Google Trends, 뉴스, DataLab 등에서 상승 키워드 탐색
- 관심도 상대순위 상위 정상 후보 25개를 최대 예비 풀로 고정
- 1위부터 순서대로 독립 조사와 Fact Ledger 기반 콘텐츠 생성
- 정상 콘텐츠 20개 확보 시 낮은 순위 미처리 후보 생략
- 성공 후보를 selectionRank 순서로 정렬해 상위 20개 공개
- 신규 성공 15~19개면 직전 정상 TOP에서 부족분 최대 5개 보충
- 신규 성공 15개 미만이거나 보충 후 20개를 만들 수 없으면 기존 TOP20 유지
- 상세 콘텐츠와 피드를 누적 보존

STELLATE는 기사 원문을 재조합하지 않습니다.
확인된 구조화 사실을 바탕으로 독립적인 설명형 콘텐츠를 작성합니다.

3. v8.0.56 수정 대상 운영 로그와 원인
======================================================================

보고된 v8.0.55 운영 결과:
- 최종 완료 25/25
- 추가 검색 처리 50
- 실행 단계 77/96
- 시작 후 54분
- 신규 정상 콘텐츠 0개
- 직전 TOP 이어받기 0개
- 대표 실패:
  영주 SK스페셜티: CONTENT_BODY_TOO_SHORT
  의협 한특위: CONTENT_BODY_TOO_SHORT
  이만희: CONTENT_BODY_TOO_SHORT
  비트코인: CONTENT_ACCURACY_FAILED
  전광훈 목사: CONTENT_BODY_TOO_SHORT

직접 원인:
1) 기존 분량 판정이 최종 피드 화면 전체가 아니라 blog 본문 문자열만 계산했음
2) 사용자가 의도한 대제목·요약·포인트·Q&A·공백이 1,000자 계산에서 빠졌음
3) AI가 피드 본문, 요약, Q&A, 인스타 카드 등을 한 번에 만들었음
4) Q&A나 인스타 카드 하나가 Fact ID·수치 검증에 걸려도 피드 전체를 폐기했음
5) 정확성 검사를 분리해도 공개 판정의 grounding·copyright 검사에 보조 콘텐츠가 포함돼 전체 후보를 다시 탈락시킬 수 있었음
6) 전체 결과가 폐기된 뒤 짧은 fallback으로 교체돼 CONTENT_BODY_TOO_SHORT가 반복됐음
7) 비트코인·인물·기관 같은 넓은 키워드는 공식 자료의 구조화 facts를 profile Fact로 충분히 재사용하지 못했음
8) 생성 순서가 사용자가 원하는 `키워드 → 피드 → 요약 → 피드 기반 제목`으로 강제되지 않았음

직전 TOP 이어받기 0개 해석:
- 신규 정상 콘텐츠가 0개이면 기존 확정 정책상 carryover를 계산하지 않는 것이 정상임
- carryover는 신규 정상 15~19개일 때 부족한 자리 최대 5개만 보충하는 정책임
- 신규 15개 미만이면 기존 TOP20 전체를 유지하며 이전 TOP 일부를 새 목록에 섞지 않음
- 따라서 이번 핵심 오류는 carryover가 아니라 신규 피드 0개였던 생성·검증 구조임

5. 최종 공개 피드 분량 계산 기준
======================================================================

최소 공개 기준: 1,000자
권장 목표: 근거가 충분하면 약 5,000자
권장 범위: 3,500~6,000자

1,000자와 5,000자 계산에 포함:
- 메인 제목
- 핵심 요약
- 맥락·왜 중요한지
- 주요 포인트
- 대제목·소제목의 화면 표시 문자열
- 본문 전체
- Q&A 질문과 답변
- 각 화면 블록 사이의 공백과 줄바꿈

계산에서 제외:
- 화면에 그대로 보이지 않는 Markdown 제어 표식 `## `
- 내부 저장용 메타데이터, Fact ID, source ID
- 인스타 카드
- 하단 출처 URL 자체

저장 필드:
- feedPublicCharacterCount: 최종 공개 텍스트 전체 길이
- feedBodyCharacterCount: blog 본문 문자열 길이
- feedLengthBreakdown.publicText
- feedLengthBreakdown.body
- feedLengthBreakdown.countingBasis

판정:
- blog 본문 단독이 1,000자 미만이어도 최종 공개 텍스트가 1,000자 이상이면 통과
- 5,000자 미달만으로 실패시키지 않음
- 6,000자 초과만으로 실패시키지 않음
- 동일 Fact 반복, 일반론, 추측, 근거 없는 전망으로 분량을 채우지 않음
- 최종 공개 텍스트가 1,000자 미만일 때만 같은 Fact 보강을 1회 수행
- 확인 Fact 자체가 부족하면 억지 통과시키지 않고 정상 실패

feedDetailLengthPolicy:
- v8056-public-text-min1000-target5000-feed-first

6. 정확성·근거·저작권 검증 분리
======================================================================

v8.0.56 feed scope:
- 피드 본문 섹션과 최종 피드 요약을 공개 후보 판단의 중심으로 사용
- 피드 본문의 claimIds, 숫자, 단위, 추론 표현을 Fact Ledger와 대조
- grounding과 copyright 판정도 contentVersion 140부터 피드 범위를 적용

보조 콘텐츠:
- Q&A와 인스타 카드는 피드 완성 후 파생
- 보조 콘텐츠에 잘못된 Fact ID나 근거 없는 표현이 있어도 검증된 피드 전체를 폐기하지 않음
- 전체 검증 모드에서는 보조 콘텐츠 오류를 별도 진단 가능
- 인스타 카드 재생성 기능의 독립 검증은 유지

유지 정책:
- 기사의 문장과 제목을 복사하지 않음
- 공식·공인·신뢰 언론 우선
- 온라인 반응은 Fact 근거로 사용하지 않음
- 분량을 위해 검증 절차 설명·출처 안내문을 본문에 넣지 않음

7. 넓은 키워드의 profile Fact 복구
======================================================================

대상 예:
- 비트코인·암호화폐·가상자산
- 인물·정치인·목사·의사
- 기관·협회·위원회·기업

수정:
- officialEvidence와 profileLinks를 함께 profile evidence로 구성
- 공식·공인·신뢰 출처의 구조화 item.facts를 profile 설명에 포함
- 주제 일치와 신뢰 출처를 확인한 기본 사실은 profile Fact로 활용
- 역할 탐지 범위를 목사, 의사, 정치인, 암호화폐, 가상자산, 기관, 협회, 위원회 등으로 확장

주의:
- profile Fact는 사건 Fact를 대신하는 일반론이 아님
- 현재 이슈 Fact가 있으면 사건 중심 피드를 우선함
- 고정 키워드 흐름에서 사건 Fact가 부족하고 신뢰 profile Fact가 있을 때만 entity profile 모드를 허용
- 실제 운영에서 넓은 키워드의 오탐·과도한 기본정보 비중을 반드시 표본 확인해야 함

8. TOP25 -> TOP20 전체 처리 흐름
======================================================================

1. Cron 또는 관리자 TOP 즉시 시작
2. runId 생성
3. 후보 약 90개 수집

6. 다음 정상 상대순위 후보로 최대 25개 고정
7. selectionRank와 candidateId 부여
8. QStash 요청당 후보 1개 처리
9. 외부 호출 전 research checkpoint와 모든 alias 저장
10. 독립 검색과 Fact Ledger 구성
11. 대표 엔티티 재검증
12. 피드 본문 우선 생성
13. 피드 요약 생성
14. 피드 기반 제목 확정
15. 보조 콘텐츠 파생
16. 최종 공개 텍스트 글자 수 계산
17. 1,000자 미만이면 same-Fact 보강 1회
18. 최종 stage·snapshot·alias 저장 및 읽기 검증
19. 정확성·분량·제목 검증
20. 실제 오류 코드 저장 후 제한 재시도
21. 정상 콘텐츠 20개 확보 시 낮은 순위 queued/retry_wait 후보를 SKIPPED_NOT_NEEDED 처리
22. cutoffRank 이내 이미 처리된 상위 retry_wait만 제한 재시도
23. 성공 후보를 selectionRank 오름차순 정렬
24. 신규 20개면 신규 상위 20개 공개
25. 신규 15~19개면 archive 포함 직전 정상 TOP에서 최대 5개 보충
26. 신규 15개 미만이거나 20개를 만들 수 없으면 기존 TOP20 유지
27. 직전 공개 TOP과 비교해 NEW·유지·상승·하락 계산
28. 최종 20개 원자적 공개

9. v8.0.55에서 유지되는 복구 정책
======================================================================

- 외부 조사·AI 호출 전 publication stage와 run snapshot에 최소 checkpoint 저장
- candidateId, slug:{slug}, stage:{stageId} alias 저장·읽기 검증
- Redis 순간 오류 시 제한 재시도와 self-healing
- 실제 실패 원인을 lastRootCauseCode에 보존
- KEYWORD_ATTEMPT_LIMIT는 attemptLimitCode로 별도 저장
- failed/private live 콘텐츠가 있으면 마지막 정상 archive로 carryover 조회
- 반도체, 가스 누출, 구조팀 같은 포괄 조각 후보 차단
- 영문·숫자가 섞인 기업명 뒤 잘못 붙은 조사 ‘서’ 제한 정규화
- 동일 canonical source URL의 동일 사건 후보 중복 제거
- 상단 `📈 TOP20`, `📋 피드` 아이콘 유지
