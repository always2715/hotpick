0. 이 파일의 역할과 우선순위
======================================================================

이 파일은 채팅방 최대 한도, 대화 소실, 담당자 변경에 대비한 단일 통합 인수인계 문서입니다.
새 채팅에서는 ZIP 안의 이 TXT 파일 하나를 먼저 읽으면 현재 STELLATE 전체 구조와 확정 정책, 오류 이력, 배포 방법, 남은 위험을 파악할 수 있어야 합니다.

우선순위:
1) 실제 최신 소스코드
2) 이 파일의 현재 최종 정책
3) README_v8.0.33.md
4) 이전 버전 README와 과거 구현 문서

과거 문서에 TOP30이 적혀 있어도 현재 공개 정책은 TOP20입니다.
과거 파일명과 Redis 키에 top30 문자열이 남아 있을 수 있으나 호환성을 위한 역사적 이름이며, 이를 근거 없이 일괄 개명하지 않습니다.

새 채팅 시작 요청 예:

ZIP 안의 STELLATE_PROJECT_HANDOFF_v8.0.33.txt를 먼저 읽고,
현재 소스와 대조해 프로젝트 상태를 파악한 뒤 이어서 개발해줘.
실제 확인하지 않은 내용은 완료로 표시하지 말고,
운영 Redis·QStash 검증이 필요한 항목은 별도로 구분해줘.

1. 프로젝트 개요
======================================================================

서비스명: STELLATE
기존 저장소/프로젝트명: hotpick
서비스 목적:
- 다양한 관심 신호에서 현재 관심도가 상승한 키워드를 선별
- 상대순위 TOP20을 먼저 고정
- 확정된 20개 키워드를 각각 독립 조사
- Fact Ledger 기반 상세 콘텐츠, 피드, 요약, 제목, 관련 뉴스, 관련 영상 생성
- TOP10은 인스타그램 카드 콘텐츠까지 생성
- 20개가 모두 준비된 경우에만 새 TOP을 한 번에 공개

STELLATE는 기사 제목·본문을 재조합하는 뉴스 복사 사이트가 아닙니다.
확인된 구조화 사실을 바탕으로 독립적인 설명형 콘텐츠를 작성해야 합니다.

현재 핵심 목표:
- TOP20이 안정적으로 끝까지 완료되는 운영 흐름
- 근거 없는 AI 문장 제거
- 기사 제목 조각이 키워드로 확정되는 문제 차단
- 피드·요약·제목이 Fact Ledger와 일치
- 관련영상은 직접 관련된 신뢰 채널 영상만 표시
- 출처 URL은 중복 없이 한 번만 표시
- 관리자에서 실행 상태·오류 위치·중단·복구 가능

5. 현재 TOP20 전체 처리 흐름
======================================================================

1. 외부 Cron 또는 관리자 `TOP 즉시 시작`
2. 실행 runId 생성
3. QStash start 메시지 등록
4. 후보 약 90개 수집
5. 키워드 정규화
6. 명백한 스팸·중복·잘린 기사 문장 조각 제거
7. 후보 풀 상대순위로 TOP20 확정
8. candidateId와 publicationStageId 부여
9. QStash batch: 요청당 후보 1개
10. 저장된 실행 원본을 먼저 확인
11. 원본이 없으면 확정 키워드만 독립 조사
12. Fact Ledger 구성
13. 상세 콘텐츠 생성·검증
14. 피드·요약·제목·관련자료 파생
15. publication stage와 run snapshot 이중 저장
16. 저장 후 다시 읽어 검증
17. 실패 후보만 추가 검색, 후보당 자동 총 3회
18. queued/processing/retry_wait가 남아 있으면 finalize 연기
19. 20개 전체 상세·피드·제목 검증
20. 피드 인덱스 검증
21. 20개를 원자적으로 공개
22. 20개 미만이면 기존 정상 TOP 유지

finalize 단계에서는 외부 검색이나 AI 생성을 다시 수행하지 않습니다.
최종 단계는 저장된 Fact Ledger와 stage만 검증·복구합니다.

6. TOP20 선정 정책
======================================================================

핵심 원칙:
- TOP 선정과 콘텐츠 검증을 분리
- 먼저 관심도 상대순위 TOP20을 확정
- 콘텐츠가 풍부해 보이는지를 TOP 통과 조건으로 사용하지 않음
- `검토기준 미충족`으로 TOP을 0~1개로 줄이지 않음
- 다만 독립 키워드가 아닌 깨진 기사 문장 조각은 확정 전에 제외

제외 대상:
- 스팸, 성인, 도박, 불법, 의미 없는 문자열
- 완전히 동일한 후보·동일 사건 중복
- 괄호가 닫히지 않은 제목 조각
- 숫자와 단위로 시작하는 문장 조각
- 서술어·조사가 남은 잘린 제목

v8.0.33 실제 실패 사례:
- `13.61포인트(1.53 오른` → 제외
- `나란히 2경기` → 제외
- `질문에 답하는` → 제외

정규화 가능 사례:
- `HBM4 출하 4개월만` → `HBM4`
- `유인영 45세 전에는 꼭` → `유인영`
- `아이폰 18 공개 소식` → `아이폰 18`

제외된 후보 자리는 다음 정상 순위 후보로 채워 TOP20을 유지합니다.

YouTube 정책:
- YouTube 단독 후보 생성 금지
- 개인 브이로그·팬 편집·재업로드 조회수만으로 TOP 후보를 만들지 않음
- 공식 채널·방송사·언론사·기관·브랜드의 동일 사건 보조 신호로만 사용

7. 콘텐츠 조사·Fact Ledger 정책
======================================================================

TOP 후보 탐색 때 사용한 기사 제목·요약·본문·링크·이미지를 상세 콘텐츠 생성 입력으로 그대로 재사용하지 않습니다.
확정 키워드를 독립적으로 조사합니다.

조사 기준:
- 최근 36시간 중심
- 인물·제품·작품 정체 확인은 공식 프로필·공식 문서 등 안정 자료 허용
- 공개 본문에는 `최근 36시간`, `36시간 내`, `36시간 기준` 표현 금지

Fact Ledger 기본 필드 개념:
- factId
- factText 또는 claim
- sourceUrl
- sourceTitle
- publisher
- publishedAt
- sourceType
- verificationStatus
- conflictGroup
- usedInSections

정확성 원칙:
- 상세 사실문은 Fact Ledger와 연결
- 날짜·금액·수치·순위·일정은 출처 확인
- 출처에 없는 숫자·날짜·원인·전망 생성 금지
- 충돌하는 사실은 확정 표현 금지
- 공식 발표와 신뢰 매체 우선
- 일반화 문구를 사실 대용으로 사용 금지

금지되는 일반화 Fact:
- 상태 변화가 확인됐습니다.
- 공식 발표가 확인됐습니다.
- 수치 변화가 확인됐습니다.
- 현재 상황이 확인됐습니다.

구체적 Fact가 같이 있으면 일반화 Fact만 제거하고 구체적 Fact로 콘텐츠를 다시 구성합니다.
구체적 Fact가 하나도 없을 때만 실패합니다.

온라인 반응:
- 블로그·카페·SNS·커뮤니티는 `온라인 동향`에만 사용
- 기본정보, 현재 이슈, Fact Ledger, 제목, 핵심 결론의 사실 근거로 사용 금지
- 일부 게시물을 전체 여론처럼 단정 금지

8. 상세·피드·요약·제목 정책
======================================================================

상세 기본 구조:
1. 키워드가 무엇인지 설명하는 기본정보
2. 현재 확인된 주요 이슈
3. 온라인 동향
4. STELLATE 인사이트
5. 연관 뉴스 최대 3건
6. 관련 영상 최대 2건
7. 출처

피드·요약:
- 상세 콘텐츠의 검증 문장만 사용
- Fact 객체 자체를 문자열 변환하지 않음
- `[object Object]` 발생 금지
- 분량을 채우기 위한 일반론 금지
- 출처에 한 문장만 확인되면 한 문장만 사용 가능
- 피드에 없는 내용을 요약에서 새로 추가 금지

제목:
- 상세 콘텐츠 완성 후 생성
- TOP 제목: `키워드 · 간단제목`
- 간단제목: 4~18자의 실제 핵심 사건
- 피드 제목: 18~32자 편집형
- 상세 제목: 28~50자 설명형
- 기사 제목 그대로 사용 금지

금지 제목:
- 수치 변화
- 상태 변화
- 시장 가격 변동
- 현재 상황
- 공식 발표
- 최신 소식
- 관심 집중
- 화제 이유

fallback:
- 기사 원제를 fallback으로 사용하지 않음
- Fact Ledger 실제 사실문에서 사건을 추출해 자연스럽게 작성

9. 관련 뉴스·영상·출처 정책
======================================================================

연관 뉴스:
- 최대 3건
- 실제 기사 제목과 실제 URL
- 동일 URL 제거
- 제목만 다른 동일 기사 제거
- 같은 사건만 반복해서 3개 채우지 않음

관련 영상:
- 최대 2건
- 키워드 또는 현재 사건과 직접 관련
- 공식 채널, 방송사, 언론사, 기관, 브랜드, 당사자 채널 우선
- 팬 편집, 개인 추측, 브이로그, 재업로드 제외
- 최대 30일까지 탐색 가능
- YOUTUBE_API_KEY가 없거나 신뢰 영상이 없으면 비워 둠
- 관련 없는 영상을 숫자 채우기 위해 넣지 않음

출처 중복:
- canonical URL 기준 한 번만 표시
- utm_source, gclid, fragment 등 추적값 제거
- 연관 뉴스에서 이미 표시한 URL은 출처 목록에서 반복 표시하지 않음
- 실제 출처 제목은 보존

23. 현재 남은 운영 확인사항
======================================================================

확실하지 않은 사실 / 운영에서 확인해야 하는 사항:
1. 실제 실패 후보가 Vercel 300초 timeout을 몇 번 발생시키는지
2. QStash가 500 응답 후 동일 cursor를 실제로 몇 회 재시도하는지
3. 운영 Redis에 이전 단계 lock이 얼마나 남아 있는지
4. Anthropic·네이버·YouTube API의 429와 지연 상태
5. v8.0.33 새 실행이 20/20까지 완료되는지

운영 재실패 시 우선 판단:
- status=processing, phase=research_and_generation, attempts 증가: 외부 조사/AI 요청 timeout 가능성
- status=retry_wait, code=INSUFFICIENT_KEYWORD_EVIDENCE: 해당 키워드 조사 근거 부족
- status=failed, code=KEYWORD_ATTEMPT_LIMIT: 3회 모두 실패
- stage_identity_mismatch: 현재 실행 후보 충돌 또는 잘못된 slug 연결
- stage_not_found + phase 미상: QStash 체인 또는 저장 전 요청 종료
- trend_step_lock_busy: 정상 재시도 유도 중이며 성공으로 소비되면 안 됨

현재 소스와 대조해서 실제 구현 상태를 확인해줘.

현재 기준:
- TOP20 원자적 공개
- 콘텐츠 버전 129
- 트렌드 캐시 49
- Redis namespace stellate:v7 유지
- QStash 요청당 후보 1개
- 후보 자동 최대 3회
- 단계 lock 240초
- 잘린 기사 제목 조각은 TOP20 확정 전 제외하고 다음 후보로 보충
- candidateId가 실행마다 바뀔 수 있으므로 실행 간 동일 slug+대표 키워드 콘텐츠는 Fact 기반 재구성 후 승격
- 현재 실행의 다른 후보 ID는 거부
- publication stage와 run snapshot candidate/slug/stage alias 이중 저장
- 기존 실패 실행은 재개하지 않고 새 실행 시작

먼저 실제 코드의 버전, QStash 흐름, Redis 키, stage 저장·조회, 후보별 phase 진단을 확인해줘.
운영 Redis·QStash 접근 없이 확인한 내용은 운영 검증 완료라고 말하지 말아줘.
수정본을 만들 경우 FULL ZIP과 GitHub Desktop REPLACE_ONLY ZIP 모두 만들고,
두 ZIP 안에 최신 전체 통합 인수인계 TXT를 반드시 포함해줘.
