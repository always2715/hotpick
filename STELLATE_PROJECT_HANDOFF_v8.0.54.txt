0. 문서 역할과 우선순위
======================================================================

이 문서는 새 채팅·담당자 변경·대화 소실 시 STELLATE를 이어가기 위한 최신 단일 인수인계 문서입니다.

우선순위:
1) STELLATE_v8.0.54_FULL.zip 실제 소스
2) 이 인수인계서의 최신 확정 정책
3) README_v8.0.54.md
4) 이전 버전 문서

실제 확인하지 않은 운영 성공은 완료로 표시하지 않습니다.
운영 Upstash Redis·QStash·Unsplash 자격증명이 필요한 검증은 로컬 테스트와 구분합니다.

1. 프로젝트 개요
======================================================================

서비스명: STELLATE
기존 저장소명: hotpick
도메인: stellate.co.kr
프레임워크: Next.js 14 Pages Router, React 18, Node.js 22.x
저장·작업: Upstash Redis, Upstash QStash
Redis namespace: stellate:v7

목적:
- Google Trends, 뉴스, DataLab 등에서 상승 키워드 탐색
- 관심도 상대순위 상위 25개를 최대 예비 후보 풀로 고정
- 1위부터 순서대로 후보별 독립 조사와 Fact Ledger 기반 콘텐츠 생성
- 정상 콘텐츠 20개 확보 시 낮은 순위의 남은 후보는 생성하지 않음
- 성공 후보를 원래 selectionRank 순서로 정렬
- 성공 후보 상위 20개 공개
- 상위 후보 실패 시 21~25위 성공 후보 승격
- 신규 성공 후보가 15~19개면 부족한 자리만 직전 정상 TOP에서 최대 5개 이어받아 TOP20 공개
- 신규 성공 후보가 15개 미만이거나 유효한 이전 콘텐츠로 20개를 채울 수 없을 때만 기존 TOP20 유지
- 상세 콘텐츠와 피드를 누적 보존

STELLATE는 기사 원문을 재조합하는 사이트가 아닙니다.
확인된 구조화 사실을 바탕으로 독립적인 설명형 콘텐츠를 작성합니다.

2-A. v8.0.50 누적 피드 별도 보존 정책
======================================================================

기존 문제:
- 과거 피드와 현재 live content가 같은 상태·인덱스에 의존함
- 이후 후보 재생성 실패, private/failed 임시 상태, 인덱스 복구 과정에서 과거 피드가 목록에서 빠질 수 있음
- 조회 fallback만으로는 source content가 이미 덮어써진 경우 마지막 정상 공개본을 보장하지 못함

v8.0.50 확정 구조:
- 정상 공개 순간 피드 목록 레코드와 상세 콘텐츠 snapshot을 별도 archive에 저장
- archive 항목은 같은 slug 기준으로 1건을 유지하며 새 TOP 작업에서 자동 삭제하지 않음
- 유지 TOP은 중복으로 계속 추가하지 않고, 새 slug가 공개될 때 누적 개수가 증가
- 현재 TOP20은 live index에서 제공
- 전체 피드와 지난 피드는 live index + cumulative archive를 병합
- live content가 이후 failed/private로 바뀌어도 관리자 공개 제한이 없으면 마지막 정상 archive content 제공
- 관리자 hidden_feed/private/trashed와 영구 삭제는 archive에도 반영
- 피드 전체 초기화는 명시적 관리자 작업이므로 archive도 삭제

주요 Redis 키:
- stellate:v7:feed:archive:items
- stellate:v7:feed:archive:contents
- stellate:v7:feed:archive:index:latest
- stellate:v7:feed:archive:slugs
- stellate:v7:feed:cumulative:migration:v8050

마이그레이션:
- 최초 피드 조회 또는 관리자 누적 피드 전체 재구성 시 기존 feedItems, legacy feed, published content:*를 archive로 이관
- 운영에서는 배포 직후 관리자 `누적 피드 전체 재구성` 1회 실행 권장
- Redis FLUSHALL 불필요 및 금지

3. v8.0.49 운영 오류와 직접 원인
======================================================================

운영 오류 예:
[top20_from25_content_incomplete]
상위 25개 중 상세·피드·제목 16/25 완료, 공개 필요 20개 미달

대표 실패:
- 이기려고 한
- 넥스트코리아 뉴김정은 본격화
- 기자수첩 인천시장직 인수위는 심판자
- AI에 돈
- 부산 어선충돌

직접 원인:
1) 관형형·서술형으로 끝난 기사 문장 조각이 후보 검증을 통과함
2) 기자수첩·사설·칼럼 같은 편집 라벨이 대표 키워드로 들어올 수 있었음
3) INSufficient evidence 오류가 재시도 사유에 포함되지 않아 일부 후보가 1회 만에 영구 실패함
4) 근거 부족 예외가 durable stage 저장 전에 발생하면 stage·snapshot·slug alias가 모두 남지 않음
5) 출처 제목에 있는 대표 엔티티를 독립 조사 검색어로 충분히 활용하지 못함
6) 이미 고정된 과거 후보 풀에는 문장 조각이 남아 있어 단순 재개로 해결되지 않음

4. v8.0.49 해결 정책
======================================================================

A. TOP25 고정 전 문장 조각 차단
- 관형형·미완결 토큰으로 끝나는 표현 차단
  예: 한, 하는, 하려고, 했던, 되는, 될, 위한, 대한, 앞둔, 본격화
- 편집 라벨 차단
  예: 기자수첩, 취재수첩, 사설, 칼럼, 인터뷰
- 숫자·괄호·서술어 중간에서 끊긴 기존 차단 규칙 유지
- 명백한 불완전 후보만 제외하며 관심도 점수 자체는 변경하지 않음
- 제외된 자리에는 다음 상대순위 정상 후보를 올려 25개 풀 유지

B. 대표 엔티티 기반 확장 조사
- 최초 발견 기사 제목과 후보 용어는 대표 엔티티를 찾는 검색 힌트로만 사용
- 최초 발견 URL·기사 본문·이미지를 사실 근거로 재사용하지 않음
- 새 검색을 독립 실행해 Fact Ledger와 근거를 구성
- 독립 조사 후 출처 제목·Fact 주체·후보 용어에서 반복 확인되는 대표 엔티티로 공개 키워드 확정
- selectionRank와 원래 관심도 상대순위는 보존

C. 근거 부족 재시도
- INSUFFICIENT_KEYWORD_EVIDENCE를 제한된 재시도 대상으로 변경
- 자동 실행 최대 3회 유지
- 관리자 명시적 재개 최대 5회 유지
- 같은 검색만 반복하지 않고 재시도 번호에 따라 대표 엔티티·사건 유형을 포함한 확장 질의 사용
- 근거가 끝내 부족하면 정상 실패하며 일반론으로 강제 통과하지 않음

D. durable 체크포인트
- 근거 부족 예외에도 research checkpoint 저장
- publication stage, run snapshot, candidate alias, slug alias를 남김
- 다음 시도와 관리자 진단에서 조사 결과와 실패 원인을 재사용 가능
- `stage·snapshot·slug alias가 모두 없음` 상태를 방지

E. 과거 실패 실행 재개 보호
- 저장된 고정 후보 25개를 관리자 재개 시 다시 검증
- 문장 조각 후보가 하나라도 있으면 `candidatePoolRebuildRequired`로 재개 차단
- 해당 실행은 중단하고 새 TOP 작업을 시작해야 함
- 정상 후보만 있는 동일 TOP25→TOP20 실행은 기존 호환 정책에 따라 재개 가능

5. TOP25 → 성공 TOP20 전체 흐름
======================================================================

1. Cron 또는 관리자 TOP 즉시 시작
2. runId 생성
3. 후보 약 90개 수집
4. 관심도 상대순위 계산
5. 스팸·중복·잘린 기사 문구·편집 라벨 제외
6. 다음 정상 상대순위 후보를 포함해 25개 고정
7. selectionRank 1~25와 candidateId 부여
8. QStash 요청당 후보 1개 처리
9. 최초 제목의 대표 엔티티 힌트 추출
10. 독립 검색과 Fact Ledger 구성
11. 조사 후 대표 엔티티 재검증
12. 상세·피드·TOP 제목 생성
13. 1,000자 미만이면 검증 Fact 기반 동일 시도 복구
14. 최종 검증 전에 stage·snapshot·alias 저장
15. 정확성·분량·제목 검증
16. 근거 부족 또는 일시 오류는 제한된 확장 조사 재시도
17. 후보 25개 상태 확정
18. 성공 후보를 selectionRank 오름차순 정렬
19. 신규 성공 후보가 20개면 앞의 20개 선택
20. 신규 성공 후보가 15~19개면 직전 정상 TOP에서 중복되지 않는 항목을 최대 5개 이어받아 20개 완성
21. 신규 성공 후보가 15개 미만이거나 유효한 이어받기 콘텐츠가 부족하면 기존 TOP 유지
22. 직전 공개 TOP20과 순위변동 계산
23. 공개 rank 1~20 부여
24. 신규 후보 + 이전 TOP 보충 항목을 20개 원자적 공개

6. 신규 진입·유지·순위변동 정책
======================================================================

비교 기준:
- 새 TOP20 공개 직전의 기존 공개 TOP20
- 생성 완료 순서나 후보 처리 순서와 비교하지 않음

동일 항목 판정 우선순위:
1) stableEventId, eventKey, trendKey
2) slug
3) 조사 후 대표 topKeyword/keyword
4) identityAliases 또는 keywordAliases

표시:
- 이전 TOP에 없음: NEW
- 동일 항목·동일 순위: 유지
- 순위 상승: ▲ 상승 폭
- 순위 하락: ▼ 하락 폭

저장 필드:
- previousRank
- rankChange
- movementStatus: new/up/down/same
- retained
- newEntry
- previousSlug
- rankIdentityScore

공개 실패 시 기존 TOP과 기존 순위변동 표시는 변경하지 않습니다.

7. 콘텐츠 분량·정확성 정책
======================================================================

피드 상세 `/feed/{slug}`:
- 최소 공개 기준 1,000자
- 사실과 근거가 충분하면 약 5,000자 권장
- 5,000자 미달만으로 실패시키지 않음
- 6,000자 초과만으로 실패시키지 않음
- 같은 Fact 반복, 근거 없는 전망·평가·추측으로 분량 채우기 금지
- 확인 사실로 1,000자를 만들 수 없으면 정상 실패

TOP 클릭 요약 `/{slug}`:
- 핵심 요약·맥락·주요 포인트 합계 최대 1,000자

Fact Ledger:
- 주체, 행동, 대상, 시점, 숫자·단위, 적용 조건 확인
- 공식·공인·신뢰 언론 우선
- 온라인 반응은 사실 근거로 사용 금지

제목:
- `대표 키워드 · 구체 사건`
- titleEvidenceFactIds 저장
- 최신 소식, 현재 상황, 관심 집중 등 포괄 제목 금지

32. v8.0.54 TOP20 조기 완료 정책
======================================================================

문제:
- TOP25가 최대 예비 후보 풀인데도 20개 정상 콘텐츠가 이미 확보된 뒤 21~25위까지 계속 생성함

확정 정책:
- generationPoolCount 25는 최대 예비 후보 한도이며 필수 완료 수량이 아님
- 후보는 selectionRank 1위부터 순서대로 처리
- 정상 콘텐츠 20개 확보 시 현재 20번째 정상 후보의 selectionRank를 cutoffRank로 확정
- cutoffRank보다 낮은 순위의 queued/retry_wait 후보는 status=skipped, errorCode=SKIPPED_NOT_NEEDED, skipReason=skipped_not_needed로 종료
- 이미 처리된 cutoffRank 이내의 retry_wait 후보는 성공 시 더 낮은 정상 후보를 밀어낼 수 있으므로 제한 재시도
- 상위 재시도가 끝나면 즉시 finalize
- skipped 후보는 실패 건수에 포함하지 않음
- 관리자 화면에는 실제 처리, 정상 준비, 불필요 미처리, 실패를 구분해 표시

예:
- 1~20위 성공: 21~25위 5개 생략
- 2위 실패 후 21위까지 처리해 정상 20개 확보: 22~25위 생략, 2위 retry_wait가 남아 있으면 해당 후보만 확인
- 25위까지 처리해 정상 19개: v8.0.51 정책에 따라 신규 19개 + 직전 TOP 1개 혼합 공개

유지 정책:
- 관심도 상대순위와 TOP25 후보 고정 방식 변경 없음
- 공개 순위는 생성 완료 순서가 아니라 selectionRank 순서
