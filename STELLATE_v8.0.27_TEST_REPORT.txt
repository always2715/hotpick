STELLATE v8.0.27 TOP REFRESH LOOP FIX TEST REPORT
Date: 2026-06-23

ROOT CAUSES CONFIRMED
1. finalizeTrendRefreshRun performed external content research for every missing candidate in up to two recovery passes.
2. Initial fixed TOP30 processing always used force:true and bypassed the valid content cache.
3. Duplicate QStash start/batch delivery did not fully protect completed positions and terminal tasks.
4. Admin progress used terminal completion rather than first-pass attempted count.

FIXED FLOW
- Initial phase: 30 candidates, 3 per QStash batch.
- Retry phase: retry_wait candidates only, 6 per QStash message, concurrency up to 3.
- Automatic attempts: maximum 2 per keyword.
- Explicit admin resume: maximum 3 per keyword.
- Finalize phase: stage validation and atomic publication only; no external search or AI generation.
- Worst-case queued step count: 17, below the configured 18-step limit.

ADDITIONAL SAFETY
- Duplicate start, batch and retry cursor protection.
- Trigger-aware QStash deduplication ID.
- Completed tasks are checked before attempt-limit handling.
- A replayed initial batch does not immediately retry retry_wait/failed/stopped tasks.
- Existing v8.0.26 batchCursor is honored.
- Stage/title/body duplicate checks are moved before bounded retry/finalize.
- First attempt may reuse valid same-keyword 12-hour content cache.
- Fact Ledger fallback accepts fact.claim and factLedger.sources.
