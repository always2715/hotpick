# STELLATE v8.0.32 — TOP20 stage 식별·이중 저장 수정

## 1. 이전 수정이 실패한 이유

v8.0.31까지의 테스트는 실제 Upstash Redis와 QStash가 연결된 운영 실행을 재현하지 못했습니다. 일부 테스트는 함수·문구 존재 여부를 확인하는 정적 검사였으며, 이를 운영 성공처럼 설명한 것은 부정확했습니다.

이번 버전에서는 반복 오류를 발생시킨 실제 코드 경로를 다시 추적했습니다.

## 2. 확인된 직접 원인

후보 검색어와 생성된 대표 키워드를 완전 일치로 비교하고 있었습니다.

예:

- 확정 후보: `HBM4 출하 4개월만`
- 생성 콘텐츠 대표 키워드: `HBM4`

두 값은 같은 주제이지만 기존 최종 검증은 문자열이 다르다는 이유로 저장된 stage를 무시했습니다. 그 결과 stage가 실제로 존재해도 `stage_not_found`로 기록됐습니다.

`유인영 45세 전에는 꼭`처럼 검색 문구가 인물명 `유인영`으로 정리되는 항목도 같은 방식으로 탈락할 수 있었습니다.

## 3. v8.0.32 수정 내용

### 3.1 콘텐츠 동일성 판정 변경

기존:

1. 후보 검색어와 콘텐츠 대표 키워드 완전 일치
2. 불일치하면 stage 무시
3. `stage_not_found` 처리

변경:

1. `candidateId` 일치 확인
2. `publicationStageId` 일치 확인
3. 레거시 데이터만 동일 slug와 키워드 포함 관계로 보조 확인
4. 다른 후보 콘텐츠만 `stage_identity_mismatch`로 분리

### 3.2 실행별 콘텐츠 이중 저장

각 완성 콘텐츠를 두 위치에 저장합니다.

- `publication_stage:{publicationStageId}`
- `cron:content:{runId}` Hash의 `{candidateId}` 필드

한쪽 저장소가 누락돼도 다른 쪽을 읽어 복구합니다.

### 3.3 생성 반환 지점에서도 즉시 이중 저장

다음 모든 경로에서 API 반환 전에 실행별 원본을 저장합니다.

- 신규 콘텐츠 생성
- 메모리 캐시 재사용
- Redis 콘텐츠 재사용
- 출처 지문 동일 콘텐츠 재사용
- Fact Ledger 기반 복구

작업 경계에서도 다시 저장·검증하여 QStash 재전달이나 서버 종료 구간의 저장 누락 가능성을 줄였습니다.

### 3.4 finalize 조기 실행 방지

최종 공개 요청이 먼저 도착해도 다음 상태가 남아 있으면 바로 실패 처리하지 않습니다.

- `queued`
- `processing`
- `retry_wait`

오래 멈춘 `processing` 항목은 재시도 대기로 전환하며, 대기 작업이 끝난 뒤에만 최종 20건 검증을 수행합니다.

### 3.5 오류 코드 정정

- 실제 저장 원본이 모두 없을 때만 `stage_not_found`
- 저장 원본은 있지만 후보 식별자가 다르면 `stage_identity_mismatch`
- 실행별 원본 저장·조회 실패는 `RUN_CONTENT_*` 코드로 분리

## 4. 버전

```json
{
  "appVersion": "8.0.32",
  "contentVersion": 128,
  "trendCacheVersion": 48,
  "publicTopCount": 20
}
```

기존 v8.0.31 실패 실행과 콘텐츠 캐시가 새 실행에 섞이지 않도록 콘텐츠·트렌드 캐시 버전을 올렸습니다.

## 5. 배포 후 실행 순서

1. GitHub Desktop에서 교체파일을 기존 `hotpick` 저장소에 덮어쓰기
2. Commit 및 Push
3. `/api/version`에서 `8.0.32`, `128`, `48` 확인
4. 관리자에서 기존 실행을 즉시 중단
5. 기존 실패 실행은 재개하지 않음
6. 새 `TOP 즉시 시작` 실행
7. `상세·피드·제목 20/20`과 공개 20건 확인

## 6. 검증 범위와 제한

확인한 항목:

- 문제 사례 `HBM4 출하 4개월만` → `HBM4` 식별 테스트
- candidateId/publicationStageId 우선 판정
- 메모리 Redis 기반 stage·run snapshot 이중 저장 및 한쪽 삭제 후 잔존 확인
- 전체 기존 회귀 테스트
- Next.js 프로덕션 빌드
- 로컬 프로덕션 서버 `/api/version` 응답

확실하지 않은 사실:

실제 운영 Upstash Redis와 QStash 자격증명에는 이 환경에서 접근할 수 없으므로, 운영 환경의 전체 TOP20 완료는 배포 후 새 실행으로 최종 확인해야 합니다. 이번에는 로컬 테스트 통과를 운영 성공으로 표현하지 않습니다.
