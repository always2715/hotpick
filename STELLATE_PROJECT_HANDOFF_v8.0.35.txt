0. 이 파일의 역할과 우선순위
======================================================================

이 파일은 채팅방 최대 한도, 대화 소실, 담당자 변경에 대비한 단일 통합 인수인계 문서입니다.
새 채팅에서는 ZIP 안의 이 TXT 하나만 먼저 읽어도 STELLATE의 목적, 확정 정책, 기술 구조, 오류 이력, 배포 방법, 현재 남은 운영 확인사항을 파악할 수 있어야 합니다.

우선순위:
1) 실제 최신 소스코드
2) 이 파일의 현재 최종 정책
3) README_v8.0.35.md
4) 이전 버전 README와 과거 구현 문서

과거 문서에 TOP30이 적혀 있어도 현재 공개 정책은 TOP20입니다.
과거 파일명과 Redis 키에 top30 문자열이 남아 있을 수 있으나 호환성을 위한 역사적 이름입니다. 근거 없이 일괄 개명하지 않습니다.

새 채팅 시작 요청 예:

ZIP 안의 STELLATE_PROJECT_HANDOFF_v8.0.35.txt를 먼저 읽고,
현재 소스와 대조해 프로젝트 상태를 파악한 뒤 이어서 개발해줘.
실제 확인하지 않은 내용을 완료라고 표시하지 말고,
운영 Redis·QStash 검증이 필요한 항목은 별도로 구분해줘.

1. 프로젝트 개요
======================================================================

서비스명: STELLATE
기존 저장소/프로젝트명: hotpick

서비스 목적:
- 다양한 관심 신호에서 현재 관심도가 상승한 키워드를 선별
- 상대순위 TOP20을 먼저 고정
- 확정된 20개 키워드를 각각 독립 조사
- Fact Ledger 기반 상세 콘텐츠, 피드, 요약, 제목, 연관 뉴스, 관련 영상 생성
- TOP10은 인스타그램 카드 콘텐츠까지 생성
- 20개가 모두 준비된 경우에만 새 TOP을 한 번에 공개

STELLATE는 기사 제목·본문을 재조합하는 뉴스 복사 사이트가 아닙니다.
확인된 구조화 사실을 바탕으로 독립적인 설명형 콘텐츠를 작성해야 합니다.

현재 핵심 목표:
- TOP20 갱신이 안정적으로 20/20 완료
- 근거 없는 AI 문장 제거
- 피드와 상세가 읽을 가치가 있는 편집형 정보 상품으로 보이게 구성
- 동일 Fact 반복과 분량 채우기 문장 차단
- 관련영상은 직접 관련된 신뢰 채널 영상만 표시
- 출처 URL은 중복 없이 한 번만 표시
- 관리자에서 실행 상태·오류 위치·중단·복구 가능

5. TOP20 전체 처리 흐름
======================================================================

1. 외부 Cron 또는 관리자 `TOP 즉시 시작`
2. 실행 runId 생성
3. QStash start 메시지 등록
4. 후보 약 90개 수집
5. 키워드 정규화
6. 스팸·중복·잘린 기사 제목 조각 제거
7. 후보 풀 상대순위로 TOP20 확정
8. candidateId와 publicationStageId 부여
9. QStash 요청당 후보 1개 처리
10. 저장된 실행 원본을 먼저 확인
11. 원본이 없으면 확정 키워드만 독립 조사
12. Fact Ledger 구성
13. 상세 콘텐츠 생성 및 정확성 검증
14. 피드·요약·제목·관련자료 파생
15. publication stage와 run snapshot 이중 저장
16. 저장 후 다시 읽어 검증
17. 실패 후보만 추가 검색, 후보당 자동 총 3회
18. queued/processing/retry_wait가 남아 있으면 finalize 연기
19. 20개 전체 상세·피드·제목 검증
20. 피드 인덱스 검증
21. 20개 원자적 공개
22. 20개 미만이면 기존 정상 TOP 유지

finalize 단계에서는 외부 검색이나 AI 생성을 다시 수행하지 않습니다.

6. TOP20 선정 정책
======================================================================

핵심:
- TOP 선정과 콘텐츠 검증 분리
- 관심도 상대순위 TOP20을 먼저 확정
- 콘텐츠 분량이 많아 보이는지를 TOP 통과 조건으로 사용하지 않음
- 독립 키워드가 아닌 깨진 기사 제목 조각은 확정 전에 제외

제외 예:
- `13.61포인트(1.53 오른`
- `나란히 2경기`
- `질문에 답하는`
- `여야 징벌적`

정규화 예:
- `HBM4 출하 4개월만` → `HBM4`
- `유인영 45세 전에는 꼭` → `유인영`
- `아이폰 18 공개 소식` → `아이폰 18`

YouTube:
- 단독 TOP 후보 생성 금지
- 개인 브이로그·팬 편집·재업로드 조회수만으로 후보를 만들지 않음
- 공식 채널·방송사·언론사·기관·브랜드의 동일 사건 보조 신호로만 사용

7. Fact Ledger와 정확성 정책
======================================================================

조사:
- 최근 36시간 중심
- 인물·제품·작품 정체 확인에는 공식 프로필·공식 문서 등 안정 자료 허용
- 공개 본문에는 조사 시간 범위 문구를 표시하지 않음

Fact Ledger 개념 필드:
- id/factId
- text/claim
- scope: profile 또는 issue
- sourceIds
- sourceType
- status
- conflicts

원칙:
- 본문 사실문은 Fact Ledger에 연결
- 숫자·날짜·금액·순위·일정은 출처 확인
- 충돌 사실은 확정 표현 금지
- 공식·공인·신뢰 언론 우선
- 온라인 반응은 사실 근거로 사용하지 않음
- AI 문안이 정확성 검사를 통과하지 못하면 검증 Fact 직접 투영 문안으로 교체
- 구체적인 Fact가 있으면 일반화 문구 때문에 후보 전체를 폐기하지 않음

금지 문구 예:
- 상태 변화가 확인됐습니다
- 수치 변화가 확인됐습니다
- 공식 발표가 확인됐습니다
- 현재 상황
- 최근 이슈

8. v8.0.35 콘텐츠 상품성 정책
======================================================================

목표는 정확성을 낮춰 문장을 화려하게 만드는 것이 아닙니다.
확인된 사실을 독자가 빠르게 이해하고, 배경·일정·실질적인 확인사항까지 파악할 수 있도록 편집하는 것입니다.

본문 구조:
- 3~5개 역할형 섹션
- 기본정보
- 지금 주목받는 단일 사건
- 사건을 이해하는 맥락·수치·일정
- 이용자가 확인할 조건·변화·다음 공식 일정
- 근거 범위 안의 STELLATE 인사이트

작성 규칙:
- PF 기본정보 Fact는 첫 섹션에만 사용
- F 현재 이슈 Fact는 사건·맥락·수치·일정·이용자 확인사항에 배치 가능
- 동일 Fact를 본문 여러 구역에서 문장만 바꿔 반복 금지
- 직접 연결되는 Fact 1~3개를 한 문단의 2~4문장으로 구성
- 문단 claimIds에 실제 사용 Fact ID 전체 연결
- 숫자·비교·일정·실질 영향은 근거가 있을 때만 별도 설명
- 근거에 없는 전망·원인·평가 추가 금지
- 분량보다 사실 밀도 우선

목표 분량:
- A: 900~1,600자
- B: 600~1,000자
- C: 320~650자
- D: 160~700자

이 범위는 기사 수나 Fact가 적은데 1,400자 이상을 강제로 채우면서 반복·일반론이 늘어나는 문제를 줄이기 위한 기준입니다.

정확성 fallback 고도화:
- 구체 Fact 문장을 자연스러운 존댓말로 통일
- 받침에 맞는 은/는, 과/와 조사 적용
- `시작된다` → `시작됩니다`, `예정이다` → `예정입니다` 등 문장 종결 정리
- 일정·수치·변경 내용에 따라 구체적인 섹션 제목 생성
- Q&A 첫 답은 프로필 Fact, 다음 답은 현재 이슈 Fact를 우선 사용

출처 URL canonical 처리:
- utm_source, utm_medium, utm_campaign 등 추적 파라미터 제거

- 콘텐츠 출처와 인프라 크레딧은 서로 다른 개념
