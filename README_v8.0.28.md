# STELLATE v8.0.28 — TOP30 28/30 반복 실패 수정

## 1. 확인된 직접 원인

v8.0.27은 각 키워드의 상세·피드·제목 stage가 정상이어도, 최종 공개 직전에 전체 본문의 토큰 유사도가 0.78 이상이면 해당 항목을 `duplicate_feed_body`로 다시 탈락시켰습니다.

STELLATE 상세 피드는 공통 섹션 구조를 사용하므로 서로 다른 키워드라도 문서 구조와 일반 설명 문장이 유사할 수 있습니다. 이 구조 유사도를 실제 콘텐츠 복제로 잘못 판단하면서 정상 stage 두 개가 빠지고 `28/30`으로 끝나는 상황이 반복될 수 있었습니다.

같은 코드의 전체 피드 품질 검사는 유사 쌍을 일부 허용하고 있었기 때문에, 개별 탈락 로직과 전체 품질 로직도 서로 모순돼 있었습니다.

## 2. 수정 내용

### 구조 유사도와 실제 복제 분리

- 단순 본문 토큰 유사도만으로 개별 TOP 항목을 탈락시키지 않음
- 키워드, fingerprint, Fact Ledger 사실, 출처 URL까지 함께 비교
- 다음 조건일 때만 심각한 실제 중복으로 판정
  - 본문 유사도 0.92 이상
  - 동일 keyword, 동일 fingerprint, 사실 중복 80% 이상 또는 출처 중복 80% 이상 중 하나 충족
- 문서 구조만 유사한 경우는 관리자 진단 경고로만 저장
- 정상 stage는 최종 공개 목록에서 제거하지 않음

### 최종 로컬 복구 1회

최종 공개 전 미완료 stage에 대해 외부 검색이나 AI 생성을 다시 실행하지 않고 다음 자료만 사용해 한 번 복구합니다.

1. 현재 실행의 publication stage
2. 같은 slug의 기존 공개 콘텐츠
3. 저장된 Fact Ledger
4. 저장된 공식·신뢰 출처

복구 가능한 항목:

- 피드 제목 형식 오류
- 피드 요약 길이 오류
- 최종 제목 검증 오류
- stage 저장 누락
- Fact ID 누락
- 충돌 사실을 본문에서 제외한 뒤 확인 사실만으로 재작성 가능한 경우

### Fact Ledger 복구 안정화

- 사실 ID가 없으면 `F1`, `F2` 형식으로 자동 부여
- `fact.text`뿐 아니라 `fact.claim`도 사용
- `sourceId`를 `sourceIds` 배열로 정규화
- 충돌 상태의 사실은 복구 본문에서 제외
- 원래 충돌 정보는 `excludedConflicts`에 보존
- 출처나 사실이 없는 콘텐츠를 임의로 생성하지 않음

### 실패 진단 개선

복구 후에도 30개가 안 되면 기존의 단순 `28/30` 메시지 대신 실제 미완료 키워드와 첫 번째 실패 사유를 함께 표시합니다.

예시:

```text
미완료 항목 — 키워드A: 확인 출처 없음 | 키워드B: stage 저장 검증 실패
```

## 3. 변경하지 않은 정책

- TOP 후보는 관심도 상대순위로 먼저 30개 확정
- 후순위 후보 교체 없음
- 관심 증가 고정문으로 숫자 채우기 없음
- 사실·출처 없는 임의 콘텐츠 생성 없음
- 30개 준비 전 부분 공개 없음
- 기존 정상 TOP30 유지
- Redis namespace `stellate:v7` 유지
- 콘텐츠 버전 `125` 유지
- 트렌드 캐시 버전 `46` 유지
- 자동 조사 1회 + 실패 항목 추가 조사 1회 유지
- finalize 단계에서 외부 검색·AI 재생성 금지

## 4. 배포 후 실행 방법

v8.0.27에서 실패한 실행은 terminal 상태이므로 그대로 재개하지 않습니다.

1. v8.0.28 배포
2. `/api/version`에서 `8.0.28` 확인
3. 관리자에 남은 실행이 동작 중이면 중단
4. `TOP 즉시 시작`으로 새 실행 생성
5. 최종 상태에서 `콘텐츠 준비 30`, `피드 준비 30`, `공개 30` 확인

Redis 초기화, `FLUSHALL`, 기존 공개 TOP 삭제는 필요하지 않습니다.

cd ".\STELLATE_v8.0.28_FULL"
