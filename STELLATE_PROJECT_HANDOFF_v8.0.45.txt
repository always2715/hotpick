======================================================================
0. 문서 역할과 우선순위
======================================================================

이 문서는 채팅 한도, 대화 소실, 담당자 변경에 대비한 최신 단일 인수인계 문서입니다.
새 대화에서는 이 파일을 먼저 읽고 실제 최신 소스와 대조해서 개발을 이어갑니다.

우선순위:
1) 실제 최신 소스코드
2) 이 문서의 현재 확정 정책
3) README_v8.0.45.md
4) 과거 버전 문서

실제 확인하지 않은 운영 결과를 완료라고 표시하지 않습니다.
운영 Upstash Redis, QStash, Unsplash 자격증명이 필요한 검증은 로컬 검증과 구분합니다.

======================================================================
1. 프로젝트 개요
======================================================================

서비스명: STELLATE
기존 저장소명: hotpick
도메인: stellate.co.kr

목적:
- Google Trends, 뉴스, 네이버 DataLab 등 관심 신호에서 상승 키워드 탐색
- 관심도 상대순위 상위 25개를 고정 생성 후보로 선정
- 각 후보를 독립 조사하고 Fact Ledger 기반 콘텐츠 생성
- 검증 성공 후보를 원래 selectionRank 순으로 정렬
- 성공 후보 상위 20개 공개
- 상위 후보 실패 시 21~25위 성공 후보 자동 승격
- 상세 콘텐츠, 누적 피드, 요약, 제목, 연관 뉴스, 관련 영상, 인스타 카드 생성
- 성공 후보가 20개 미만이면 새 결과를 부분 공개하지 않고 기존 정상 TOP 유지

STELLATE는 기사 원문을 조합하거나 단순 재작성하는 사이트가 아닙니다.
확인된 구조화 사실을 바탕으로 독립적인 설명형 콘텐츠를 작성합니다.

3. v8.0.45 핵심 장애 원인과 해결
======================================================================

운영 증상:
- 피드 목록은 표시되지만 현재 공개 TOP20만 나타남
- 새 TOP 갱신 뒤 이전 피드가 다음 페이지에 누적되지 않는 것처럼 보임

확인된 직접 원인:
- lib/kv.js의 기본 최신순 조회가 최신 피드 인덱스에서 target 수량을 찾으면 즉시 반환했음
- target 기본값이 페이지 크기 20이어서 현재 TOP20이 있으면 과거 공개 콘텐츠 병합 단계가 실행되지 않았음
- Redis에 과거 content 원본이 남아 있어도 전체 feed scope에서 읽지 못했음

v8.0.45 해결:
- scope=all에서는 최신 인덱스 20건만으로 조기 반환하지 않음
- content index와 `stellate:v7:content:*` 키를 SCAN해 과거 공개 원본을 발견
- 발견한 공개 콘텐츠를 피드 인덱스에 복구
- 현재 TOP·누적 공개 콘텐츠·스냅샷·레거시 피드를 slug 기준 병합
- 전체 병합 후 정렬·필터·페이지네이션
- 관리자 `누적 피드 전체 재구성`으로 과거 content 원본 전체 복구
- 원자적 TOP 공개는 신규 20건을 추가/갱신할 뿐 기존 누적 피드를 삭제하지 않음

4. 누적 피드 최종 정책
======================================================================

피드는 현재 TOP 목록이 아니라 누적 공개 콘텐츠 저장소입니다.

scope별 정책:
- scope=all: 현재 TOP + 과거 공개 피드 전체
- scope=top: 현재 공개 TOP20만
- scope=past: 현재 TOP을 제외한 과거 공개 피드

정상 조회 순서:
1) 피드 ZSET/Hash 인덱스 조회
2) 콘텐츠 인덱스 조회
3) 필요 시 `stellate:v7:content:*` 원본 키 SCAN
4) 현재 TOP 상세 콘텐츠 원본 조회
5) TOP 스냅샷과 레거시 피드 조회
6) slug 기준 병합·중복 제거
7) visibility/status/category/search/scope 필터
8) latest/oldest/views/sequence 정렬
9) 전체 total 계산
10) offset/limit 페이지네이션

누적 보존 원칙:
- 새 TOP 공개 시 과거 feedLatest/feedItems/content 원본을 삭제하지 않음
- 같은 slug는 새 내용으로 갱신하되 중복 행을 만들지 않음
- 과거 공개 콘텐츠는 contentVersion이 이전 버전이어도 숨기지 않음
- private, trashed, hidden_feed는 누적 피드에서 제외
- Redis FLUSHALL과 피드 초기화 금지

복구:
- 관리자 `누적 피드 전체 재구성`
- 과거 공개 content 원본을 최대 5,000건까지 탐색
- 누락된 content index와 feed Hash/ZSET/category/sequence/views 인덱스 재구성
- AI 호출 없이 기존 원본을 우선 복구
- 현재 TOP 중 상세 원본 자체가 없을 때만 별도 생성 대상이 될 수 있음

주요 Redis 키:
- stellate:v7:content:*
- stellate:v7:content:index
- stellate:v7:feed:items
- stellate:v7:feed:index:latest
- stellate:v7:feed:index:sequence
- stellate:v7:feed:index:views
- stellate:v7:feed:slugs
- stellate:v7:feed:cumulative:migration:v8044

5. 콘텐츠 길이 정책
======================================================================

화면 역할을 명확하게 분리합니다.

A. 피드 상세 `/feed/[slug]`
- contentVersion 135부터 본문 최소 1,000자
- 확인된 Fact와 출처가 충분하면 약 5,000자를 권장
- 일반적인 권장 범위는 3,500~6,000자이나 하드 성공 조건이 아님
- 1,000자 이상이면 5,000자 미달만으로 공개 실패시키지 않음
- 6,000자 초과도 사실 반복이 없다면 분량만으로 실패시키지 않음
- 근거 없는 일반론·추측·전망 또는 같은 Fact 반복으로 분량을 채우지 않음
- 확인 사실만으로 1,000자를 구성하지 못하면 가짜 문장 대신 후보 실패 처리
- 기존 공개 콘텐츠는 길이 정책 때문에 피드에서 삭제하지 않음

B. TOP 클릭 요약 `/{slug}`
- 요약(summary), 맥락(why), 주요 포인트(points) 합계 최대 1,000자
- 신규 생성 카드뿐 아니라 기존 저장 콘텐츠도 렌더링 단계에서 최대 1,000자로 제한
- 문장 경계 우선 축약
- 요약 화면에서 피드 상세 전체 본문을 노출하지 않음
- 상세 읽기는 `/feed/[slug]`로 연결

길이 기준 상수:
- FEED_DETAIL_MIN_CHARS = 1000
- FEED_DETAIL_TARGET_CHARS = 5000
- FEED_DETAIL_RECOMMENDED_MIN_CHARS = 3500
- FEED_DETAIL_RECOMMENDED_MAX_CHARS = 6000
- TOP_PREVIEW_MAX_CHARS = 1000

주요 파일:
- lib/feedLengthPolicy.js
- lib/topPreviewPolicy.js
- lib/contentGrade.js
- lib/contentArchitecture.js
- lib/publicationPolicy.js
- lib/api.js
- lib/feedFirstPipeline.js
- pages/[slug].js

6. TOP25 → 성공 TOP20 정책
======================================================================

1) 후보 약 90개 수집
2) 스팸·중복·잘린 문구 제거
3) 관심도 상대순위 계산
4) 상위 25개 고정 생성 풀 확정
5) selectionRank 1~25 저장
6) 후보당 독립 조사·Fact Ledger·콘텐츠 생성
7) 후보당 자동 최대 3회 제한 재시도
8) 25개 상태 확정
9) 성공 후보를 selectionRank 오름차순 정렬
10) 앞의 20개 선택
11) 공개 rank 1~20 재부여
12) 원래 순위는 selectionRank/sourceRank로 보존
13) 원래 21~25위 공개 시 promotedFromReserve=true
14) 성공 후보 20개 이상일 때만 원자적 공개
15) 20개 미만이면 기존 TOP 유지

v8.0.45에서 변경하지 않은 항목:
- 후보 수집과 관심도 점수
- 상대순위와 중복 통합
- generationPoolCount=25
- publicTopCount=20
- reserve promotion
- 원자적 공개 조건
- 이미지 점수와 TOP 점수 분리

최소 1,000자·약 5,000자 권장 정책은 TOP 관심도 순위를 바꾸지 않습니다.
콘텐츠 검증에 실패하면 고정 후보 풀 내 다음 순위 성공 후보가 승격합니다.

7. 키워드·제목·콘텐츠 정확성
======================================================================

키워드:
- 검색 문구를 그대로 공개 키워드로 사용하지 않음
- 조사 후 Fact Ledger와 공인 출처에서 공통 확인되는 대표 엔티티 사용
- 잘린 기사 제목 조각, 관형어·서술어 중간 표현, 수치로 시작하는 조각 제외

제목:
- TOP 제목: 대표 키워드 · 구체 사건
- Fact Ledger의 현재 사건 Fact 우선
- titleEvidenceFactIds 저장
- 포괄 제목 금지: 최신 소식, 현재 상황, 확인된 변화, 관심 집중 등

본문:
- 3~5개 역할형 섹션
- 주체·행동·대상·시점·수치·조건 명확화
- 동일 Fact 반복 금지
- 기사 원문 재작성 금지
- 근거 없는 전망·원인·결과 추가 금지
- 온라인 반응은 사실 근거로 사용하지 않음
