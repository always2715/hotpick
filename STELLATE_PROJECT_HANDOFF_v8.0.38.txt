0. 이 파일의 역할과 우선순위
======================================================================

이 파일은 채팅방 최대 한도, 대화 소실, 담당자 변경에 대비한 단일 통합 인수인계 문서입니다.
새 채팅에서는 ZIP 안의 이 TXT 하나만 먼저 읽어도 STELLATE의 목적, 현재 정책, 기술 구조,
오류 이력, 배포 방법, 테스트 범위, 남은 운영 확인사항을 파악할 수 있어야 합니다.

우선순위:
1) 실제 최신 소스코드
2) 이 파일의 현재 최종 정책
3) README_v8.0.38.md
4) 이전 버전 README와 과거 구현 문서

중요 용어:
- 공개 TOP 수: 20개
- 생성·검증 후보 풀: 관심도 상대순위 상위 25개
- 과거 파일명과 Redis 키에 top30 문자열이 남을 수 있으나 역사적 호환 이름임
- 근거 없이 파일명·Redis namespace·key를 일괄 개명하지 않음

새 채팅 시작 요청 예:

ZIP 안의 STELLATE_PROJECT_HANDOFF_v8.0.38.txt를 먼저 읽고,
현재 소스와 대조해 프로젝트 상태를 파악한 뒤 이어서 개발해줘.
실제 확인하지 않은 내용을 완료라고 표시하지 말고,
운영 Redis·QStash 검증이 필요한 항목은 별도로 구분해줘.

1. 프로젝트 개요
======================================================================

서비스명: STELLATE
기존 저장소/프로젝트명: hotpick
서비스 도메인: stellate.co.kr

서비스 목적:
- Google Trends, 뉴스, DataLab 등 여러 관심 신호에서 상승 키워드 탐색
- 관심도 상대순위 상위 25개를 생성 후보 풀로 고정
- 25개 후보를 각각 독립 조사하고 Fact Ledger 기반 콘텐츠 생성
- 검증 성공 후보를 원래 순위대로 정렬해 상위 20개만 공개
- 실패한 상위 후보가 있으면 21~25위 중 다음 성공 후보 자동 승격
- 상세 콘텐츠, 피드, 요약, 제목, 연관 뉴스, 관련 영상 생성
- 공개 TOP10에는 인스타그램 카드 콘텐츠 생성
- 성공 후보가 20개 미만이면 새 결과를 부분 공개하지 않고 기존 TOP20 유지

STELLATE는 기사 제목·본문을 조합하는 뉴스 복사 사이트가 아닙니다.
확인된 구조화 사실을 바탕으로 독립적인 설명형 콘텐츠를 작성해야 합니다.

현재 핵심 목표:
- TOP 집계 방식은 기존 상대순위 기준 유지
- 생성 실패가 일부 있어도 25개 풀에서 성공한 20개 공개
- 잘린 검색 문구가 아닌 확인된 대표 엔티티를 키워드로 사용
- 제목은 핵심 사건 Fact와 연결하고 포괄 제목 차단
- 피드·상세는 정확성과 상품성을 함께 확보
- 근거 없는 AI 문장, 동일 Fact 반복, 분량 채우기 일반론 제거
- 관련영상은 직접 관련된 신뢰 채널 영상만 표시
- 출처 URL은 중복 없이 한 번만 표시
- 관리자에서 실행 상태·중단·복구·오류 원인 확인

2-A. v8.0.38 대표 이미지·썸네일 최종 정책
======================================================================

핵심 원칙:
- TOP 선정·관심도 점수·후보 순위·승격·원자적 공개와 썸네일 선택을 완전히 분리
- 이미지 유무, 이미지 적합도, Unsplash API 상태를 TOP 점수에 절대 사용하지 않음
- 썸네일 저장소/API 오류는 콘텐츠 생성·검증·TOP 공개 실패로 전파하지 않음

허용 출처:
- TOP, 피드, 상세, 인스타 카드 대표 이미지는 Unsplash만 허용
- 공식 사이트 이미지, 언론사 OG/기사 이미지, YouTube 썸네일은 대표 이미지 후보에서 완전 제외
- 허용 이미지 호스트는 images.unsplash.com
- imageMeta.source는 unsplash만 허용
- 관련 YouTube 링크·제목·채널·날짜는 유지하지만 영상 썸네일은 표시하지 않음

사전 이미지 풀:
- 10개 카테고리 × 10개 슬롯, 총 100개
- 카테고리: 경제·금융, 기업·산업, IT·AI, 사회·생활, 정치·행정, 국제·글로벌, 문화·연예, 스포츠, 건강·교육, 환경·재난
- 각 슬롯에 내부 ID, 분위기 제목, 분위기 태그, 소재 태그, 톤, 적합/회피 유형 저장
- 운영 UNSPLASH_ACCESS_KEY로 관리자에서 한 번 구축하고 Upstash Redis에 저장
- 100개를 모두 확보하지 못하면 부분 풀을 저장하지 않음
- 자동 구축 결과는 automated-screened 상태이며 관리자가 표본 검수·수정·비활성화 가능

선택 순서:
1. 상세 콘텐츠와 제목 생성 완료
2. 대표 카테고리 판정
3. 분위기·소재·긍정/중립/부정 톤 판정
4. 카테고리 +50, 분위기 최대 +20, 소재 최대 +15, 톤 +10, 적합 유형 +10
5. 회피 유형 -50, 최근 사용 -30, 과다 사용 감점, 같은 화면 중복 강한 제외
6. 가장 높은 점수의 활성 이미지를 선택
7. 선택 결과를 thumbnailImageId 중심으로 콘텐츠에 고정 저장

지속성과 반복 방지:
- 동일 이미지는 최근 20개 신규 배정에서 우선 제외
- 같은 화면에서 이미 사용한 ID는 신규 배정 후보에서 제외
- 대안이 없을 때만 최근 사용 제한을 완화

- 이미지 파일이 실제로 표시 불가능하거나 관리자가 직접 교체한 경우에만 변경

5. v8.0.37 TOP25 → 성공 TOP20 전체 처리 흐름
======================================================================

1. 외부 Cron 또는 관리자 `TOP 즉시 시작`
2. 실행 runId 생성
3. QStash start 메시지 등록
4. 후보 약 90개 수집
5. 스팸·중복·잘린 기사 제목 조각 제거
6. 관심도 상대순위 계산
7. 상위 25개를 고정 생성 풀로 확정
8. 후보마다 selectionRank 1~25와 candidateId 부여
9. QStash 요청당 후보 1개 처리
10. 저장된 실행 원본과 stage 먼저 확인
11. 원본이 없으면 확정 후보를 독립 조사
12. 조사 후 확인된 대표 엔티티로 공개 키워드 검증
13. Fact Ledger 구성
14. 상세 콘텐츠 생성 및 정확성 검증
15. 피드·요약·TOP 제목·피드 제목·상세 제목 생성
16. publication stage와 run snapshot 이중 저장
17. 저장 후 다시 읽어 검증
18. 실패 후보만 후보당 자동 최대 3회 제한 재시도
19. queued/processing/retry_wait가 남아 있으면 finalize 연기
20. 25개 후보의 완료·실패 상태 수집
21. 성공 후보를 selectionRank 오름차순 정렬
22. 성공 후보 중 앞의 20개 선택
23. 공개 rank는 1~20으로 재부여
24. 원래 순위는 selectionRank/sourceRank로 보존
25. 원래 21~25위가 공개되면 promotedFromReserve=true
26. 공개 대상 20개의 피드 품질과 저장 상태 검증
27. 20개 원자적 공개
28. 성공 후보가 20개 미만이면 기존 공개 TOP 유지

중요:
- 21~25위는 실행 중 새로 찾는 대체 후보가 아니라 시작 시 고정된 예비 후보
- 실패 후보를 임의로 관심 증가 고정문이나 가짜 콘텐츠로 채우지 않음
- 최종 공개 단계에서는 외부 검색이나 AI 재생성을 하지 않음
- 25개 전체를 조사한 뒤 성공 순위 기준으로 20개 선택

예:

고정 후보 순위: 1~25
실패: 2위, 7위, 18위
성공 후보 순서: 1, 3, 4, 5, 6, 8 ... 21, 22, 23
공개: 성공 후보 중 원래 순위가 높은 20개

6. TOP 후보 선정 정책
======================================================================

핵심:
- TOP 선정과 콘텐츠 품질 검증 분리
- 관심도 상대순위 방식 유지
- 절대 품질 점수 때문에 후보 풀 자체를 0~1개로 줄이지 않음
- 상위 25개를 고정하고 콘텐츠 성공 여부는 이후 단계에서 판단

제외 대상:
- 스팸·성인·도박·불법·무의미 문자열
- 동일 엔티티·동일 사건 중복
- 닫히지 않은 괄호
- 수치·단위로 시작하는 기사 제목 조각
- 관형어·서술어 중간에서 끊긴 표현

과거 제외 예:
- `13.61포인트(1.53 오른`
- `나란히 2경기`
- `질문에 답하는`
- `여야 징벌적`

조사 후 정규화 예:
- `HBM4 출하 4개월만` → `HBM4`
- `유인영 45세 전에는 꼭` → `유인영`
- `아이폰 18 공개 소식` → `아이폰 18`

YouTube:
- YouTube 단독 TOP 후보 생성 금지

- 공식 채널·방송사·언론사·기관·브랜드의 동일 사건 보조 신호로만 사용

7. 키워드 정확성 정책
======================================================================

후보 검색 문구를 그대로 공개 키워드로 확정하지 않습니다.
독립 조사 후 다음 자료에서 공통으로 확인되는 대표 엔티티를 사용합니다.

- Fact Ledger의 주체
- 공식·공인 출처 제목
- 신뢰 언론의 사건 주체
- 제품·작품·정책의 정식 명칭

대표 키워드 변경 조건:
- 후보 문구에 대표 엔티티가 포함됨
- Fact Ledger 또는 출처 제목에서 같은 엔티티가 확인됨
- 다른 동명이인·다른 제품으로 바뀌지 않음
- 지나친 축약으로 사건 주체가 사라지지 않음

저장 필드:
- topKeyword
- keywordVerification
- candidateId
- selectionRank

금지:
- 기사 문장 조각을 키워드로 사용
- 검색 결과 첫 제목만 보고 엔티티 확정
- 동명이인 결합
- 제품명과 기업명을 근거 없이 교체

8. Fact Ledger와 정확성 정책
======================================================================

조사 범위:
- 최근 36시간 중심
- 인물·제품·작품 정체 확인에는 공식 프로필·공식 문서 등 안정 자료 허용
- 공개 본문에는 `최근 36시간` 같은 내부 조사 범위 문구를 표시하지 않음

Fact Ledger 개념 필드:
- id/factId
- text/claim
- scope: profile 또는 issue
- sourceIds
- sourceType
- status
- conflicts

원칙:
- 본문 사실문은 Fact Ledger와 연결
- 숫자·날짜·금액·순위·일정은 출처 확인
- 충돌 사실은 확정 표현 금지
- 공식·공인·신뢰 언론 우선
- 온라인 반응은 사실 근거로 사용하지 않음
- AI 문안이 정확성 검사를 통과하지 못하면 검증 Fact 직접 투영 문안으로 교체
- 구체 Fact가 있으면 일반화 Fact 한 줄 때문에 전체 후보를 폐기하지 않음

v8.0.37 추가 검증 단위:
- 주체(subject)
- 행동(action)
- 대상(object)
- 시점(time)
- 숫자와 단위(number/unit)
- 적용 조건(condition)

금지 문구 예:
- 상태 변화가 확인됐습니다
- 수치 변화가 확인됐습니다
- 공식 발표가 확인됐습니다
- 현재 상황
- 최근 이슈

10. 피드·상세 콘텐츠 작성 정책

목표:

- 확인된 사실을 독자가 빠르게 이해하도록 편집
- 사건, 배경, 수치, 일정, 실제 확인사항을 구분
- 같은 Fact 반복과 분량 채우기 일반론 제거

본문 구조:
- 3~5개 역할형 섹션
- 무엇인지 또는 누구인지
- 지금 주목받는 단일 사건
- 사건을 이해하는 배경·수치·일정
- 이용자가 확인할 조건·변화·다음 공식 일정
- 근거 범위 안의 STELLATE 인사이트

v8.0.37 문장 검증:
- 문장에 주체가 명확해야 함
- `그`, `이것`, `해당`, `관련`처럼 대상이 모호한 표현으로 시작 금지
- 동명이인·다른 기업·다른 경기·다른 작품을 혼합 금지
- 발표일, 시작일, 종료일, 예정일을 시간 순서에 맞게 작성
- 수치와 단위는 Fact Ledger 값을 보존
- 근거 없는 원인·결과·평가·전망 추가 금지
- 독자 행동은 예매, 신청, 출시, 공식 일정처럼 출처가 있는 정보만 사용

동일 Fact 사용:
- 한 Fact는 본문 한 역할 구역에만 배치
- 직접 연결된 Fact 1~3개는 한 문단의 2~4문장으로 연결 가능
- 문단 claimIds에 실제 사용 Fact ID 전체 연결

권장 분량:
- A: 900~1,600자
- B: 600~1,000자
- C: 320~650자
- D: 160~700자

분량이 아니라 사실 밀도를 우선합니다.

13. 관련 뉴스·영상·출처 정책

연관 뉴스:
- 최대 3건
- 실제 제목과 실제 URL
- 동일 URL·동일 사건 중복 제거
- 키워드·사건과 직접 관련된 자료만 표시

관련 영상:
- 최대 2건
- 영상 ID 또는 직접 URL 중 하나가 있으면 표시 가능
- 공식·방송·언론·기관·브랜드·당사자 채널 우선
- 팬 편집·개인 추측·재업로드 제외
- 직접 관련 영상이 없으면 억지로 채우지 않음

출처 canonical 처리:
- utm_source, utm_medium, utm_campaign 제거
- gclid 등 광고 추적값 제거
- fragment 제거
- 같은 canonical URL은 한 번만 표시
- 연관 뉴스에 표시된 URL은 자료 출처에서 다시 표시하지 않음
- 실제 자료 제목, 언론사/기관명, 날짜 유지 가능

온라인 반응:
- 온라인 동향 구역 참고 정보로만 사용
- 기본정보·이슈·요약·인사이트·결론의 사실 근거로 사용 금지
- 일부 게시물을 전체 여론처럼 단정 금지

Upstash 표시:
- 공개 페이지 하단에 작은 인프라 크레딧
- `STELLATE 데이터 저장·작업 큐`
- `Powered by Upstash Redis · QStash`
- 콘텐츠 출처와 인프라 크레딧은 서로 다른 개념
